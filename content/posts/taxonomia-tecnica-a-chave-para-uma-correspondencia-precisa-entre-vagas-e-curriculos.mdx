## Introdução

Imagine que você está desenvolvendo um sistema para conectar talentos a oportunidades de emprego. A tarefa de comparar um currículo com uma vaga parece direta: basta verificar se as tecnologias listadas no currículo correspondem às exigidas na vaga. No entanto, essa simplicidade rapidamente se desfaz quando nos deparamos com a variedade de formas como uma mesma tecnologia pode ser mencionada. Uma vaga pode pedir `Node.js`, outra `NodeJS`, e um currículo pode listar `Vue.js` enquanto a vaga apenas menciona `frontend JavaScript`. Como garantir que essas correspondências sejam feitas corretamente?

## O problema

A inconsistência na nomenclatura de tecnologias e habilidades é um obstáculo significativo em sistemas de correspondência automática entre currículos e vagas. Essa variação ocorre devido a diferentes convenções de escrita, abreviações, erros de digitação e a forma como os dados são extraídos de fontes distintas.

Por exemplo, um sistema pode extrair "JavaScript" de um currículo e, ao comparar com uma vaga que lista "JavaScript", falhar na correspondência simplesmente porque a forma como a informação foi processada difere. Isso leva a falsos negativos (tecnologias que existem, mas não são reconhecidas) e, consequentemente, a uma análise de compatibilidade imprecisa. A falta de um padrão comum para representar termos técnicos gera ruído e diminui a confiabilidade do sistema.

## A solução

A abordagem mais eficaz para mitigar essas inconsistências é a implementação de uma **taxonomia técnica centralizada**. Essa taxonomia atua como um dicionário mestre, definindo termos canônicos (a forma "oficial" ou preferencial de um termo) e mapeando todos os seus possíveis aliases e variações.

Ao criar uma taxonomia, podemos estruturar os termos por categorias, como linguagens de programação, frameworks frontend, frameworks backend, bancos de dados, etc. Cada termo canônico terá uma lista associada de todos os seus sinônimos e variações que podem aparecer em currículos ou vagas.

Veja um exemplo conceitual de como essa taxonomia pode ser estruturada:

```php
<?php

class TechnicalKeywordTaxonomy
{
    public function terms(): array
    {
        return [
            'programming_languages' => [
                'javascript' => ['javascript', 'js'],
                'typescript' => ['typescript', 'ts'],
                'python' => ['python'],
                'php' => ['php'],
                'go' => ['golang', 'go developer', 'go backend'],
            ],
            'frontend_frameworks' => [
                'vue' => ['vue', 'vue.js', 'vuejs'],
                'react' => ['react', 'react.js', 'reactjs'],
                'angular' => ['angular'],
            ],
            'backend_frameworks' => [
                'laravel' => ['laravel'],
                'django' => ['django'],
                'nestjs' => ['nestjs', 'nest.js', 'nest js'],
            ],
            'databases' => [
                'postgresql' => ['postgresql', 'postgres'],
                'mysql' => ['mysql'],
            ],
            'devops' => [
                'ci_cd' => ['ci/cd', 'continuous integration', 'continuous deployment'],
            ],
            'apis' => [
                'rest_api' => ['rest api', 'restful api'],
            ],
        ];
    }
}
```

Com essa estrutura, a lógica de comparação pode ser simplificada: todos os termos extraídos (sejam de currículos ou vagas) são normalizados para seus valores canônicos antes de serem comparados. Isso garante que `Node.js`, `NodeJS` e `node js` sejam todos tratados como o termo canônico `javascript` (ou `nodejs`, dependendo da granularidade desejada na taxonomia).

## Exemplo prático

A extração de termos técnicos de um texto pode ser realizada percorrendo a taxonomia. Para cada termo encontrado no texto, verificamos se ele corresponde a algum alias definido. Se sim, o termo canônico associado é registrado.

```php
<?php

class TermExtractor
{
    private TechnicalKeywordTaxonomy $taxonomy;

    public function __construct(TechnicalKeywordTaxonomy $taxonomy)
    {
        $this->taxonomy = $taxonomy;
    }

    public function extract(string $text): array
    {
        $normalizedText = strtolower($text); // Normaliza o texto de entrada
        $foundCanonicalTerms = [];

        foreach ($this->taxonomy->terms() as $category => $terms) {
            foreach ($terms as $canonical => $aliases) {
                foreach ($aliases as $alias) {
                    // Usa expressão regular para encontrar o alias como palavra inteira
                    if (preg_match('/\b' . preg_quote($alias, '/') . '\b/i', $normalizedText)) {
                        $foundCanonicalTerms[] = $canonical;
                        break 2; // Encontrou um alias para este termo canônico, passa para o próximo termo canônico
                    }
                }
            }
        }

        return array_values(array_unique($foundCanonicalTerms));
    }
}
```

Uma vez que os termos de um currículo (`resumeSignals`) e de uma vaga (`jobKeywords`) são normalizados e extraídos usando a taxonomia, a comparação se torna robusta:

```php
// Supondo que $jobKeywords e $resumeSignals já foram extraídos e normalizados para termos canônicos
$matched = array_intersect($jobKeywords, $resumeSignals);
$missing = array_diff($jobKeywords, $resumeSignals);
```

Isso garante que a comparação entre `Node.js` (vaga) e `nodejs` (currículo) resulte em um match positivo.

## Decisões técnicas

A adoção de uma taxonomia centralizada traz diversas vantagens:

*   **Consistência:** Garante que todos os termos técnicos sejam representados de forma uniforme, independentemente da fonte.
*   **Confiabilidade:** Reduz drasticamente a ocorrência de erros de correspondência devido a variações de nomenclatura.
*   **Manutenibilidade:** Facilita a atualização e expansão do sistema. Adicionar uma nova tecnologia ou um novo alias é tão simples quanto atualizar a taxonomia.
*   **Clareza:** Simplifica a lógica de comparação, pois a complexidade das variações é encapsulada na taxonomia.

Além disso, é crucial considerar a **força do sinal** de cada termo. Nem todos os termos têm o mesmo peso. `Python` ou `Laravel` são indicadores fortes de habilidade, enquanto termos como `API` ou `Data` podem ser contextuais e aparecer em diversas funções. Uma estratégia para lidar com isso é classificar os termos na taxonomia (e.g., `strong`, `contextual`, `weak`) e ajustar a lógica de análise com base nessa classificação.

Por exemplo, um currículo que lista apenas termos contextuais genéricos pode ter sua qualidade de análise limitada, enquanto um que lista termos fortes específicos terá uma análise mais confiável.

```php
enum SignalStrength: string
{
    case Strong = 'strong';
    case Contextual = 'contextual';
    case Weak = 'weak';
}
```

Outra decisão importante é a **supressão de termos genéricos**. Quando um termo específico está presente, ele deve ter precedência sobre termos mais genéricos. Por exemplo, `REST API` é mais informativo que apenas `API`. Da mesma forma, `Data Engineering` ou `Machine Learning` são mais específicos que apenas `Data`. A taxonomia pode ser usada para definir essas relações de hierarquia ou substituição, garantindo que a análise foque nos termos mais relevantes.

## O que dá para melhorar

Embora a taxonomia resolva muitos problemas, há nuances a serem consideradas:

*   **Ambiguidade de termos:** Alguns termos podem ter múltiplos significados. Por exemplo, `go` pode se referir à linguagem de programação Go ou ao verbo "ir". Para mitigar isso, os aliases na taxonomia devem ser mais específicos, como `golang`, `go developer`, `go backend`, para garantir que o contexto técnico seja capturado.
*   **Contexto para termos genéricos:** A classificação de termos como "contextual" é um bom começo, mas a lógica de análise pode se beneficiar de regras mais sofisticadas para interpretar a relevância de termos contextuais com base em outros termos presentes.
*   **Manutenção da taxonomia:** À medida que o ecossistema tecnológico evolui, a taxonomia precisará ser atualizada. Definir um processo claro para adicionar novos termos, aliases e categorias é essencial para a longevidade do sistema.
*   **Tratamento de habilidades "soft":** A taxonomia apresentada foca em habilidades técnicas. A integração de habilidades comportamentais (soft skills) exigiria uma estrutura complementar.

## Conclusão

A implementação de uma taxonomia técnica centralizada é uma estratégia poderosa para construir sistemas de correspondência de vagas e currículos mais precisos e confiáveis. Ao normalizar termos, gerenciar aliases e considerar a força e relevância dos sinais, superamos as inconsistências inerentes à linguagem natural e garantimos que as conexões entre talentos e oportunidades sejam feitas com maior eficácia. Essa abordagem não só melhora a qualidade da análise, mas também simplifica a evolução e manutenção do sistema a longo prazo.
