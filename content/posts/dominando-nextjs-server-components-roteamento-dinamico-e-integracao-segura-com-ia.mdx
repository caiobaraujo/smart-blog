## Introdução

Ao iniciar um novo projeto com Next.js, especialmente após as recentes atualizações que priorizam Server Components por padrão, é comum se deparar com desafios inesperados. A transição de paradigmas e a necessidade de adaptar abordagens antigas podem gerar dúvidas e exigir um entendimento mais profundo das novas funcionalidades. Este artigo aborda os aprendizados e as melhores práticas ao estruturar um projeto Next.js, focando em Server Components, roteamento dinâmico e a integração segura de serviços externos, como APIs de Inteligência Artificial.

## O problema

A principal dificuldade encontrada ao começar um novo projeto Next.js, sob a ótica dos Server Components por padrão, é a adaptação do fluxo de desenvolvimento. Tentar aplicar padrões de interatividade, como o uso de `useState` diretamente em uma página, pode levar a erros inesperados no console, pois a interatividade e hooks como `useState` ou `onClick` exigem a marcação explícita do componente como um Client Component através da diretiva `'use client'`. Além disso, a configuração de rotas dinâmicas e a forma de realizar fetching de dados também mudaram significativamente, afastando-se de práticas comuns como o uso de `useEffect` para buscas iniciais. A integração com serviços externos, como APIs de IA, levanta questões de segurança cruciais, especialmente no que diz respeito à exposição de chaves de API.

## A solução

A estratégia adotada envolve abraçar o modelo de Server Components para tarefas pesadas, como o fetching de dados, e utilizar Client Components apenas para funcionalidades que demandam interatividade. Essa abordagem otimiza a performance, pois minimiza a quantidade de JavaScript enviada ao navegador do usuário. O roteamento dinâmico é configurado de forma eficiente, utilizando a estrutura de pastas do Next.js para mapear URLs a componentes específicos. Para o fetching de dados, a função do componente é definida como `async`, permitindo a chamada direta de funções de busca de dados no servidor, eliminando a necessidade de `useEffect` para a carga inicial e evitando o "flicker" de carregamento. A segurança na integração com APIs externas é garantida pela criação de rotas de API no backend do Next.js, que atuam como um proxy seguro para as chaves de API.

## Exemplo prático

### Configurando Roteamento Dinâmico

Para criar rotas dinâmicas, como para um blog, a estrutura de pastas do Next.js é fundamental. Um arquivo como `app/blog/[slug]/page.tsx` permite capturar o valor do `slug` da URL.

```bash
mkdir -p "app/blog/[slug]"
touch "app/blog/[slug]/page.tsx"
```

No arquivo `page.tsx`, o `slug` pode ser acessado através dos `params`:

```tsx
// app/blog/[slug]/page.tsx
async function getPostBySlug(slug: string) {
  // Lógica para buscar o conteúdo do post (ex: de um arquivo Markdown)
  // Retorna os dados do post
  return {
    title: `Título do Post ${slug}`,
    content: `Conteúdo do post ${slug}...`,
  };
}

export default async function BlogPostPage({ params }: { params: { slug: string } }) {
  const post = await getPostBySlug(params.slug);

  return (
    <div>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </div>
  );
}
```

### Integração Segura com API de IA

Para integrar com uma API de IA, como o Gemini, a chave de API nunca deve ser exposta no frontend. Uma rota de API no Next.js é a solução ideal.

```tsx
// app/api/generate/route.ts
import { GoogleGenerativeAI } from '@google/generative-ai';

const API_KEY = process.env.GOOGLE_AI_STUDIO_API_KEY;

if (!API_KEY) {
  throw new Error("Chave de API do Google AI Studio não configurada.");
}

const genAI = new GoogleGenerativeAI(API_KEY);

export async function POST(request: Request) {
  const { text } = await request.json();

  if (!text) {
    return new Response(JSON.stringify({ error: "Texto não fornecido" }), { status: 400 });
  }

  try {
    const model = genAI.getGenerativeModel({ model: "gemini-pro" });
    const result = await model.generateContent(text);
    const response = result.response;
    const generatedText = response.text();

    return new Response(JSON.stringify({ generatedText }), { status: 200 });
  } catch (error) {
    console.error("Erro ao gerar conteúdo com Gemini:", error);
    return new Response(JSON.stringify({ error: "Falha ao gerar conteúdo" }), { status: 500 });
  }
}
```

No frontend, um botão pode disparar uma requisição para esta rota de API. É importante gerenciar um estado de `isGenerating` para desabilitar o botão durante o processamento da IA.

```tsx
// Exemplo conceitual de frontend
'use client';

import { useState } from 'react';

export default function GeneratorComponent() {
  const [inputText, setInputText] = useState('');
  const [generatedContent, setGeneratedContent] = useState('');
  const [isGenerating, setIsGenerating] = useState(false);

  const handleGenerate = async () => {
    setIsGenerating(true);
    try {
      const response = await fetch('/api/generate', {
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
        },
        body: JSON.stringify({ text: inputText }),
      });
      const data = await response.json();
      setGeneratedContent(data.generatedText);
    } catch (error) {
      console.error("Erro ao chamar API de geração:", error);
    } finally {
      setIsGenerating(false);
    }
  };

  return (
    <div>
      <textarea value={inputText} onChange={(e) => setInputText(e.target.value)} />
      <button onClick={handleGenerate} disabled={isGenerating}>
        {isGenerating ? 'Gerando...' : 'Gerar Conteúdo'}
      </button>
      {generatedContent && <p>{generatedContent}</p>}
    </div>
  );
}
```

## Decisões técnicas

A adoção de Server Components por padrão no Next.js 13+ é uma decisão arquitetural que visa melhorar a performance e a experiência do desenvolvedor. Ao manter o fetching de dados e a lógica de renderização no servidor, reduzimos o tempo de carregamento inicial e a quantidade de JavaScript no cliente. A necessidade de usar `'use client'` para componentes interativos é uma clareza explícita sobre onde a interatividade reside, facilitando o raciocínio sobre o fluxo de dados e a execução do código. A criação de rotas de API para interagir com serviços externos é uma prática de segurança fundamental, encapsulando credenciais sensíveis e expondo apenas a funcionalidade necessária ao frontend.

## O que dá para melhorar

Embora a abordagem de Server Components e Client Components seja poderosa, a gestão de estado entre eles pode se tornar complexa em aplicações maiores. A estratégia de fetching de dados assíncrono diretamente nos Server Components é eficiente, mas para cenários que exigem revalidação de dados em tempo real ou cache mais granular, bibliotecas como React Query ou SWR podem ser consideradas, embora sua integração com Server Components exija um planejamento cuidadoso. A experiência do usuário durante o processamento de IA pode ser aprimorada com indicadores de progresso mais visuais ou a utilização de WebSockets para atualizações em tempo real, caso a API de IA suporte.

## Conclusão

Estruturar um projeto Next.js moderno com Server Components, roteamento dinâmico e integração segura de APIs é um passo essencial para construir aplicações web performáticas e escaláveis. Compreender as nuances entre Server e Client Components, adotar práticas de fetching de dados otimizadas e priorizar a segurança ao lidar com chaves de API são pilares para o sucesso no desenvolvimento com o framework.
